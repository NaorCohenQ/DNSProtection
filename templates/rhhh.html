Hierarchical Heavy Hitters (HHH) Algorithm

The Hierarchical Heavy Hitters (HHH) algorithm is designed to detect traffic sources responsible for generating large amounts of data across different hierarchical levels, such as IP address ranges (/8, /16, /24, /32).

It identifies heavy hitters within a hierarchy by analyzing traffic patterns at multiple granularities. For example, it helps determine which IP prefixes are generating the most traffic, making it useful in detecting potential congestion or DDoS attacks.

Randomized Hierarchical Heavy Hitters (RHHH) Algorithm

The Randomized Hierarchical Heavy Hitters (RHHH) algorithm extends the HHH algorithm by introducing randomization to improve scalability and efficiency. In environments where tracking large volumes of traffic is computationally expensive, RHHH optimizes detection by using random sampling while maintaining accuracy.

RHHH works by sampling traffic at various hierarchical levels (e.g., /8, /16, /24) to track the most significant traffic sources. It’s particularly useful for identifying DDoS attack sources or anomalies in high-traffic environments.

  • Tracks significant traffic sources across multiple hierarchical levels (e.g., IP prefixes).
  • Randomized sampling improves performance without sacrificing accuracy.
  • Scales to handle high-traffic environments.
  • Helps detect potential DDoS attacks and other network anomalies in real-time.

Our DNS Attack Detection Algorithm

Our algorithm is designed to detect and block DNS attacks, particularly focusing on NXDOMAIN (NXD) attacks. The algorithm combines:

  • Dynamic IP behavior monitoring using RHHH: The RHHH algorithm, particularly its update method, tracks patterns from various IP addresses and monitors the volume of traffic.
  • Threshold-based Blocking: IP addresses that seem to be part of an attack are blocked based on a dynamically adjusted threshold, calculated the used traffic patterns tracked by RHHH.

Our algorithm works as follows:

  1. The algorithm receives a packet and maintains a blacklist of blocked IP addresses.
  2. If the packet’s source IP is in the blacklist, the request is blocked immediately.
  3. If the IP is not on the blacklist, the packet is sent to the DNS server. This action either succeeds or fails:
    • If the request succeeds (i.e., it’s legitimate), the IP is passed to the update function of RHHH, and its traffic is tracked as legitimate.
    • If the request fails (NXDOMAIN), we maintain a counter K for that specific IP to track how many times the DNS server couldn't process the request.
  4. If K exceeds a threshold Y (calculated from RHHH counters for the IP), the IP is added to the blacklist.
  5. Blocked IPs stay on the blacklist for a certain number of ticks and are automatically removed afterward. If an IP reappears on the blacklist during this period, it is blocked immediately.

Example Scenario

Scenario: An IP sends 10 legitimate DNS requests followed by 50 NXDOMAIN requests.

  • For each legitimate request, the RHHH algorithm updates its counters, recording the IP as legitimate traffic.
  • As NXDOMAIN requests begin arriving, the counter K for this IP increments.
  • Once K exceeds the threshold Y (calculated from RHHH counters for the IP), the IP is added to the blacklist.
  • Subsequent requests from this IP are blocked automatically, preventing further attack attempts.

By combining dynamic traffic monitoring and threshold-based blocking, this approach effectively mitigates DNS attacks, ensuring legitimate traffic flows while identifying and blocking potential attackers.
